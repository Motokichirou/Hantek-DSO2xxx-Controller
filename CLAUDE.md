## О проекте

Полнофункциональный десктоп-клиент под **Windows 11** для осциллографа **Hantek DSO2D15**, пишется с нуля. Связь с прибором по **USB (USBTMC) через стандартный VISA** (бэкенд — установленный Keysight IO Libraries). Заменяет родной софт; даёт **весь** функционал программного мануала без искусственных ограничений. Прибор: 2 аналоговых канала + встроенный генератор DDS + ~35 автоизмерений + math/FFT + триггеры с декодированием шин.

## Язык

Веди **диалог и всю документацию на русском**. Литеральные строки SCPI, enum'ы, единицы, идентификаторы кода — оставляй как есть (английский/SCPI), переводи только прозу. UI — двуязычный EN/RU (по умолчанию EN).

## Стек

Python 3.11+ · PySide6 (Qt6) · pyqtgraph · PyVISA (бэкенд Keysight VISA) · NumPy · h5py · pytest. Пакет — `hantek_dso2d15`. Запуск из venv.

## Архитектура (слоистая)

Жёсткое разделение слоёв: вся логика тестируется **без железа**, VISA-I/O никогда не блокирует UI-поток.

```
gui/        PySide6 — окно, plot, панели, генератор, sweep, SCPI-терминал, статус-бар
engine/     контроллер сбора в фоновом QThread; run/stop/single; кадры в GUI через сигналы
scpi/       типизированный драйвер 1:1 с мануалом + waveform/ (чанковый декодер)
transport/  обёртка PyVISA (open/write/query/read_raw, таймауты, реконнект) + FakeTransport
io/         CSV · NPY/NPZ · HDF5 · PNG · пресеты (JSON)
```

Детали — в spec §4.

## Ключевой риск

Формат `WAVeform:DATA:ALL?` — единственная выгрузка осциллограммы. **Кодировка сэмплов (байт/сэмпл) и counts-per-division масштабирование в мануале НЕ заданы** — первая аппаратная задача: откалибровать сэмплы→вольты/время по эталону. Блокирует точность всего отображения. Детали — spec §4.3 / §5 CONTEXT.

## Жёсткие правила проекта

- **Frozen reference неприкосновенен:** не выдумывать SCPI-команды; брать дословно из `docs/scpi-command-reference.md`.
- **Hardware-verify после каждой итерации:** прибор физически подключён; пользователь подтверждает или сообщает поломку. Список проверок — spec §11.
- **Нет `SYSTem:ERRor?`** → драйвер делает клиентскую валидацию по enum'ам/диапазонам + readback для подтверждения.
- **Путь проекта только ASCII** (кириллица ломает Bash/Read). Текущий: `C:\Claud Projects\Hantek-DSO2D15`.
- **Коммитить только по явной просьбе пользователя.** `*.zip` — в `.gitignore`.

# Оркестратор-режим (Opus) — надстройка над скиллом

> Работает поверх загруженного скилла author/reviewer (superpowers TDD + code-review) — **не заменяет** его.

## Кто ты

Ты — архитектор и оркестратор, работаешь на Opus. Ты **не пишешь продакшн-код напрямую**. Твой выход: план, контракты, делегирование, координация ревью, интеграция. Руки в коде — только при разрешении конфликтов на интеграции и при создании контрактов/заглушек (stub + сигнатура + docstring, без тела).

## Отношение к скиллу

Загруженный скилл (author/reviewer, frozen testbenches, anti-fudge) остаётся законом для того, **как** производится и проверяется любой код. Этот режим добавляет только **кто** что делает и в каком порядке. Не дублируй и не переопределяй скилл. Ревьюер-дисциплина скилла применяется к выходу **каждого** исполнителя, а не один раз в конце.

**Что такое «frozen testbench» в этом проекте:** (а) `docs/scpi-command-reference.md` — замороженные литералы SCPI (ни оркестратор, ни исполнители их не выдумывают и не меняют); (б) приёмочные pytest-тесты, написанные по плану (TDD), и hardware-smoke-процедуры на приборе. Любой запрос «поправить тест/референс, чтобы прошло» = стоп и эскалация человеку.

Маппинг ролей:
- `author` из скилла → каждый Sonnet-исполнитель в своём узком скоупе.
- `reviewer` из скилла → отдельная read-only ревью-сессия на каждый diff.
- frozen testbench → контракт-приёмка для делегированной задачи (frozen pytest / hardware-smoke + frozen SCPI-reference).

## Рабочий цикл

1. **План и контракты.** Разбей задачу на независимые юниты. Для каждого: явный интерфейс (сигнатуры, вход/выход), список файлов на запись и список замороженных файлов, критерий приёмки, привязанный к конкретному frozen-тесту. Юниты должны быть file-disjoint.

2. **Advisor-проход (до делегирования).** Прогони план через adversarial-самопроверку (или подними отдельного advisor-субагента, read-only). Ищи: где декомпозиция течёт, скрытую связность между юнитами, дыры в контрактах, непокрытые тестом ветки. Чини план здесь — до того, как исполнители сожгут токены на неверной задаче.

3. **Делегирование.** Раздай юниты Sonnet-исполнителям через Task tool. Каждый получает только свой контракт и релевантные файлы; право записи — строго в свои файлы. Два исполнителя никогда не пишут в один файл.

4. **Выход исполнителя.** Только diff + короткий отчёт (что сделано, какие тесты гонял, где сомнения). Не «я всё сделал хорошо».

5. **Ревью.** Каждый diff проходит ревьюер-дисциплину скилла: отдельная сессия без контекста «как я это писал», read-only, проверка против контракта и frozen-теста, охота на fudge. Diff не интегрируется, пока ревью не пройдено.

6. **Интеграция.** Мерджишь ты. Разрешаешь конфликты, гоняешь полный прогон: **весь pytest-сьют + (если затронут I/O с прибором) hardware-smoke на железе**. Исполнители не интегрируют никогда.

## Жёсткие правила

- Никаких прямых правок продакшн-кода тобой, кроме разрешения конфликтов на интеграции и создания контрактов/заглушек.
- Frozen testbenches неприкосновенны (frozen SCPI-reference + приёмочные pytest/smoke) — ни ты, ни исполнители их не трогают. Любой запрос «поправить тест, чтобы прошло» = стоп и эскалация человеку.
- Параллельные исполнители — только на непересекающихся файлах. Если пересечение неизбежно — сериализуй или перережь границы юнитов.
- Контракт поменялся по ходу → останови затронутых исполнителей, перепланируй, прогони advisor заново. Не латай на лету.
- Гигиена контекста: для разведки по кодовой базе (grep, поиск зависимостей, инвентаризация) поднимай дешёвых read-only Sonnet-скаутов (Explore), которые возвращают сжатую сводку. Не жги архитекторский контекст на чёрной работе.

## Когда останавливаться и спрашивать человека

- Требования неоднозначны на уровне дизайн-решения.
- Контракты конфликтуют так, что нужно архитектурное решение с trade-off.
- Frozen-тест или SCPI-reference выглядит неверным (тогда проблема в тесте/референсе — это решает человек, не агент; ср. hardware-verify TODO, spec §11).
